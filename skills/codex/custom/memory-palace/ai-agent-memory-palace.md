# AI/Agent 记忆宫殿方法

## 1. 核心定位

人类记忆宫殿依赖空间、图像和情绪冲击。大模型和 Agent 没有真实空间感，也不会像人一样长期保存画面，因此不能照搬人的记忆宫殿。

适合大模型的记忆宫殿，本质是：

> 用固定结构承载知识，用稳定路径组织检索，用强关联减少遗忘和混淆，用 Markdown 形成可复用外部记忆。

它不是“脑内永久记住”，而是把上下文、知识、任务经验和决策依据写成可被后续 Agent 快速读取、检索、续作的结构化记忆文件。

## 2. 适用对象

这套方法适用于：

- ChatGPT、Claude、Gemini、DeepSeek、Qwen 等大模型
- Codex、Cursor、Claude Code 等代码 Agent
- 工作流 Agent、数据分析 Agent、文档 Agent
- 多 Agent 协作中的共享记忆
- 项目长期演进中的决策、规则、经验沉淀

## 3. 总原则

### 3.1 记结构，不记流水账

记忆文件不记录所有聊天过程，只记录未来会影响判断和行动的信息。

优先保留：

- 目标
- 约束
- 决策
- 规则
- 偏好
- 关键事实
- 可复用流程
- 踩坑经验
- 待办和未决问题

避免保留：

- 临时寒暄
- 已完成且无复用价值的过程
- 重复描述
- 没有结论的碎片
- 过期状态

### 3.2 古典核心：地点、图像、顺序、醒目、巡行

记忆宫殿的原典核心来自古典 method of loci：人工记忆由地点和图像构成。

对 AI/Agent 记忆宫殿来说：

- 地点：稳定的九区结构、目录、索引和 L0-L3 读取层级。
- 图像：具体的记忆卡片、标题、标签、触发场景、风险条件和行动提示。
- 顺序：固定路线让 Agent 能正向、反向或从中间接续检索。
- 醒目：卡片必须具体、可触发、可区分，不能写成模糊摘要。
- 巡行：审计、复盘和下次启动提示相当于反复走过宫殿以刷新记忆。

因此，写记忆不是把信息塞进文件，而是把信息放到稳定位置，并赋予它能被再次唤起的形状。

### 3.2.1 双语原典和资料层

本体系同时保留英文古典原典和中文记忆术资料：

- 英文原典层：用于追溯 method of loci 的早期系统表达，重点吸收地点、图像、顺序、醒目和训练原则。
- 中文资料层：用于补充中文术语、中文表达和实用训练方法，帮助 Agent 在中文项目中更自然地解释和执行记忆宫殿。

吸收规则：

- 原文或扫描资料先进入资料层，不直接改写为规范结论。
- 只有经过 OCR、人工校对或可靠摘录的内容，才可写入主规范。
- 每次吸收必须保留书名、路径、页码或章节线索。
- 中英文术语发生冲突时，优先保留并列映射，不强行合并。

### 3.3 原文层和索引层分离

长期记忆应区分两类内容：

- 原文层：用户明确表达、关键决策、重要文档、代码证据、测试输出和参考资料。
- 索引层：记忆卡片、标签、触发场景、路径、摘要和下次启动提示。

索引层用于快速定位和判断，不应伪装成完整原文。对于会影响后续决策的信息，记忆卡片应尽量保留来源文件、路径、命令、日期或原文位置。

适用规则：

- 重要用户偏好、数据口径、架构决策和安全规则，应保留可追溯来源。
- 摘要可以帮助检索，但不能替代原始证据。
- 不确定的摘要必须标为 `draft` 或 `#需复查`。
- 自动压缩时只压缩索引层，不删除仍有价值的原文层。

### 3.4 本地优先和增量写入

默认把 Markdown、项目文件和用户明确提供的本地资料作为记忆基础。外部项目只通过链接、说明或独立路径引用；除非用户明确要求，不把外部仓库复制进本 SKILL，也不依赖外部 API 才能读取核心记忆。

写入应优先增量追加或局部更新：

- 不用破坏性重建来替代审计和修复。
- 覆盖旧规则前必须保留替代关系。
- 删除、归档和迁移都要留下原因。
- 自动化写入必须能被后续 Agent 审计。

### 3.5 先定位，再写入

每条记忆必须知道自己属于哪个“宫殿区域”。

不要把所有内容堆到一个长列表里。应先判断它是项目目标、用户偏好、系统规则、领域知识、代码经验，还是任务进度。

### 3.6 每条记忆都要可检索

每条记忆应包含：

- 标题
- 标签
- 触发场景
- 具体内容
- 使用方式
- 更新时间

Agent 后续读取时，能快速判断“这条记忆什么时候该用”。

### 3.7 记忆要有生命周期

记忆不是越多越好。每条记忆应标记状态：

- `active`：当前有效
- `draft`：待确认
- `deprecated`：已废弃
- `superseded`：被新规则替代
- `archived`：仅保留历史参考

### 3.8 冲突时保留来源和优先级

当新记忆和旧记忆冲突时，不直接覆盖，应记录：

- 新规则是什么
- 旧规则是什么
- 为什么替换
- 替换日期
- 谁触发了替换

## 4. AI 记忆宫殿结构

建议每个记忆文件都按以下九个区域组织。

### 4.1 入口：身份与目标

记录这个记忆文件服务于什么对象、项目或长期目标。

适合记录：

- 项目名称
- 用户长期目标
- Agent 角色
- 当前阶段
- 成功标准

### 4.2 门厅：全局规则

记录所有后续任务都应遵守的稳定规则。

适合记录：

- 用户偏好
- 输出风格
- 技术栈约束
- 禁止事项
- 文件组织方式
- 验证标准

### 4.3 地图：知识索引

记录知识分区和文件链接，让 Agent 快速找到上下文。

适合记录：

- 相关文档
- 关键目录
- 入口文件
- 数据源
- API 文档
- 设计稿位置

### 4.4 展厅：核心概念

记录领域模型、核心术语、业务规则和概念关系。

适合记录：

- 术语解释
- 实体关系
- 业务流程
- 重要公式
- 关键指标口径

### 4.5 工坊：操作流程

记录可重复执行的方法。

适合记录：

- 开发流程
- 部署流程
- 测试流程
- 数据处理步骤
- 文档生成步骤
- 排障步骤

### 4.6 档案室：决策记录

记录已经做过的重要选择，以及选择原因。

适合记录：

- 架构决策
- 技术选型
- 产品取舍
- 字段定义
- 接口约定
- 兼容性判断

### 4.7 警示墙：风险与坑

记录容易犯错、已经踩过坑、后续必须小心的地方。

适合记录：

- 易错命令
- 兼容性问题
- 数据质量问题
- 边界条件
- 性能风险
- 安全风险

### 4.8 工作台：当前任务状态

记录正在推进的任务，不用于永久知识沉淀。

适合记录：

- 当前目标
- 已完成
- 进行中
- 阻塞点
- 下一步
- 负责人或 Agent 分工

### 4.9 出口：复盘与下次启动提示

记录下一次 Agent 接手时最应该先读的内容。

适合记录：

- 最近一次工作摘要
- 下次启动步骤
- 当前最重要判断
- 需要用户确认的问题

## 5. 单条记忆卡片格式

每条记忆建议使用以下格式：

```markdown
### M-YYYYMMDD-序号：一句话标题

- 状态：active
- 区域：门厅/地图/展厅/工坊/档案室/警示墙/工作台/出口
- 标签：#项目 #领域 #技术 #偏好
- 触发场景：什么时候应该想起这条记忆
- 记忆内容：具体事实、规则或经验
- 使用方式：后续 Agent 应如何应用
- 来源：用户明确要求/代码观察/测试结果/项目文档/推理结论
- 更新时间：YYYY-MM-DD
```

## 6. 记忆写入流程

每次需要形成记忆时，按以下步骤处理。

### 6.1 提取

从对话、代码、文档或执行结果中提取可能有长期价值的信息。

判断标准：

- 未来是否会再次影响决策
- 是否能减少重复解释
- 是否能避免重复踩坑
- 是否属于用户稳定偏好
- 是否属于项目稳定约束

### 6.2 分类

把信息放入九个区域之一。

如果一条信息能放多个区域，优先级如下：

1. 全局规则
2. 决策记录
3. 风险与坑
4. 操作流程
5. 核心概念
6. 知识索引
7. 当前任务状态
8. 复盘提示
9. 身份与目标

### 6.3 压缩

把长内容压缩成可复用的原则或步骤。

推荐结构：

```text
事实：发生了什么
判断：这意味着什么
行动：以后应该怎么做
```

### 6.4 关联

为记忆增加标签、触发场景和相关文件。

Agent 检索时主要依靠：

- 标题
- 标签
- 触发场景
- 区域
- 文件路径

### 6.5 写入

写入对应区域，并保持格式统一。

如果新记忆替代旧记忆，要把旧记忆状态改为 `superseded`，并在新记忆中引用旧记忆编号。

### 6.6 复查

写入后检查：

- 是否过于啰嗦
- 是否有明确使用场景
- 是否和旧记忆冲突
- 是否包含敏感信息
- 是否应该放到另一个文件

## 7. 记忆检索流程

Agent 接到新任务时，按以下顺序读取记忆：

1. 入口：确认目标和角色
2. 门厅：读取全局规则
3. 工作台：确认当前状态
4. 地图：找到相关文件
5. 展厅/工坊/档案室/警示墙：按任务类型读取
6. 出口：读取上次交接提示

不同任务的优先读取区域：

- 写代码：门厅、地图、工坊、警示墙、工作台
- 改产品方案：入口、展厅、档案室、工作台
- 查问题：地图、警示墙、工坊、档案室
- 写文档：门厅、展厅、档案室、出口
- 接续任务：工作台、出口、警示墙、地图

## 8. 分层唤醒协议

借鉴 MemPalace 的 L0-L3 wake-up 思路，Agent 不应每次读取全部记忆，而应按任务需要分层加载。

### 8.1 L0：身份和边界

始终读取：

- 入口：项目、用户、Agent 角色和成功标准
- 门厅：全局规则、用户偏好、禁止事项和安全边界

目标：在最少上下文里知道“我是谁、为谁做、不能越过什么线”。

### 8.2 L1：当前故事

通常读取：

- 工作台：当前目标、已完成、阻塞点、下一步
- 出口：最近交接、未决问题、下次启动提示

目标：在接续任务时不重新猜当前状态。

### 8.3 L2：按需区域

按任务读取：

- 地图：需要找文件、命令、目录、数据源时
- 展厅：需要理解术语、业务、指标、领域模型时
- 工坊：需要执行流程、开发、测试、部署、排障时
- 档案室：需要理解历史决策和取舍时
- 警示墙：需要避坑、审计、安全检查、发布前确认时

目标：只加载对当前任务有用的区域。

### 8.4 L3：深度原文

必要时读取：

- 原始项目文件
- 历史归档
- 参考仓库
- 命令输出
- 长文档、日志和测试报告

目标：当索引层不够可靠或需要精确证据时，回到原文层。

### 8.5 唤醒预算

默认启动时只读 L0 和 L1。只有当任务需要时再进入 L2 或 L3。

如果读取内容超过当前上下文需要，应先生成临时任务摘要，不把所有原文塞进对话。

## 9. 多 Agent 协作规则

### 9.1 一个 Agent 写，其他 Agent 读

同一时间尽量只有一个 Agent 负责更新记忆文件，避免冲突。

### 9.2 每次更新必须留下摘要

更新记忆后，在“出口”区域补充本次变更摘要。

### 9.3 分工要写入工作台

多 Agent 并行时，应记录：

- Agent 名称或角色
- 负责范围
- 不应修改的范围
- 当前状态
- 交付物

### 9.4 不确定信息必须标记

推理出来但未验证的信息，状态设为 `draft`，来源写 `推理结论，待验证`。

## 10. 细节图表生成协议

细节点流程图、架构图、系统图、云架构图、泳道图、ER 图等，应参考 `next-ai-draw-io` 的思路：优先生成可编辑图表，而不是只生成静态图片。

### 10.1 产物选择

按需求选择图表格式：

- draw.io XML：适合复杂架构图、流程图、云架构图、需要长期维护和手工编辑的图。
- Mermaid：适合轻量说明、文档内嵌、简单流程、状态流和快速预览。
- 静态图片：适合视觉展示、封面、概念插画，不适合作为可维护架构图主产物。

默认规则：

- 用户说“细节点”“架构图”“流程图”“以后还要改”“draw.io”“可编辑”时，优先 draw.io XML。
- 用户只需要在 Markdown 中快速说明时，可以用 Mermaid。
- 如果使用云厂商、Kubernetes、BPMN、网络设备等专业图标，应先查 shape library 或参考库，不猜图标语法。

### 10.2 生成前规划

生成复杂图前，先写 2-3 句规划：

- 图表目的：说明什么系统、流程或关系。
- 布局方向：左到右、上到下、泳道、分层、中心辐射或网格。
- 分组方式：用户层、接入层、服务层、数据层、外部系统、运维监控等。
- 关键连接：主链路、回调、异常路径、数据流、控制流。
- 校验点：不重叠、不越界、箭头不穿节点、文字可读、可继续编辑。

### 10.3 draw.io 生成规则

draw.io 图表应遵守：

- 使用稳定、唯一、可读的 cell id。
- 节点、容器、连接线分层组织。
- 连接线明确 source、target、entry/exit 点。
- 复杂连接使用 waypoints 绕开中间节点。
- 保持单页视口优先，避免元素过分分散。
- 文字短而明确，长说明放到图外文档。
- 不把图表语义只藏在颜色里，标签必须能独立说明含义。

### 10.4 编辑已有图

修改已有图时：

1. 先读取当前 XML、节点 ID 或现有结构。
2. 小改动使用 ID 级 update/add/delete。
3. 不因局部修改重建整张图，避免覆盖用户手工调整。
4. 重大重构前先说明会替换整体结构。
5. 每次修改前尽量保留版本或导出备份。

### 10.5 图表校验

交付前至少检查：

- XML 可解析，没有重复 ID、未闭合标签、非法实体、嵌套 mxCell。
- 连接线 source/target 存在。
- 节点不重叠，箭头不穿过非相关节点。
- 文字在节点内可读，不遮挡。
- 图表能在 draw.io/diagrams.net 中打开和继续编辑。
- 如果导出 PNG/SVG，应同时保留可编辑源文件。

### 10.6 记忆写入

图表相关记忆应记录：

- 图表文件路径
- 源格式：draw.io XML/Mermaid/SVG/PNG
- 生成依据：需求、代码、接口、数据库、用户说明
- 主要布局规则
- 校验结果
- 下次编辑注意事项

## 12. 推荐文件命名

按用途拆分记忆文件：

```text
memory/
  00-index.md                  # 总索引
  01-user-preferences.md       # 用户稳定偏好
  02-project-memory.md         # 项目长期记忆
  03-domain-knowledge.md       # 领域知识
  04-engineering-decisions.md  # 技术和架构决策
  05-task-state.md             # 当前任务状态
  06-risk-log.md               # 风险和踩坑记录
```

如果项目较小，也可以只用一个文件：

```text
doc/ai-memory.md
```

## 13. 标准标签体系

标签用于跨项目检索，必须稳定、短小、可组合。不要为每条记忆临时发明过细标签。

### 13.1 标签维度

每条记忆建议使用 2 到 5 个标签，优先从以下维度选择：

- 项目类型：`#数据`、`#前端`、`#后端`、`#全栈`、`#移动端`、`#文档`、`#研究`、`#运维`
- 记忆区域：`#目标`、`#规则`、`#索引`、`#概念`、`#流程`、`#决策`、`#风险`、`#状态`、`#交接`
- 技术主题：`#SQL`、`#API`、`#数据库`、`#测试`、`#部署`、`#权限`、`#性能`、`#安全`
- 治理状态：`#待确认`、`#已验证`、`#已废弃`、`#已替代`、`#需复查`
- 协作场景：`#用户偏好`、`#Agent协作`、`#多项目`、`#迁移`、`#复盘`

### 13.2 标签规则

- 标签使用中文优先，技术缩写可保留英文大写，如 `#API`、`#SQL`。
- 标签不要写成长句，长解释放在触发场景或记忆内容里。
- 同义标签只保留一个，例如统一用 `#风险`，不要混用 `#坑`、`#警告`、`#注意`。
- 项目专属标签可以添加，但必须能被后续 Agent 理解。

### 13.3 标签示例

```markdown
- 标签：#数据 #SQL #流程 #已验证
- 标签：#前端 #API #风险 #需复查
- 标签：#文档 #决策 #用户偏好
```

## 14. 文件拆分与归并阈值

记忆文件需要在“集中”和“可维护”之间平衡。太集中会难检索，太分散会难接续。

### 14.1 保持单文件的条件

满足以下条件时，可以继续使用单文件：

- 记忆卡片少于 30 条
- 只有一个主要项目或主题
- 当前只有一个 Agent 或一个人维护
- 工作台内容每次都能在 3 分钟内读完

### 14.2 拆分为多文件的条件

出现以下任一情况，应考虑拆分：

- 记忆卡片达到 30 条以上
- 工作台历史开始淹没长期规则
- 一个文件同时承载多个项目
- 风险、决策或流程需要频繁独立检索
- 多 Agent 并行时容易修改同一个文件

### 14.3 推荐拆分顺序

优先按以下顺序拆分：

1. `05-task-state.md`：把当前状态从长期记忆中分离。
2. `06-risk-log.md`：把风险和踩坑记录独立出来。
3. `04-engineering-decisions.md`：把决策记录独立出来。
4. `03-domain-knowledge.md`：把领域概念和业务规则独立出来。
5. `01-user-preferences.md`：把用户稳定偏好独立出来。

### 14.4 归并条件

如果某个文件 30 天以上没有更新，且内容少于 5 条，可归并回项目主记忆或归档。

### 14.5 拆分后的索引要求

拆分后必须在 `00-index.md` 或总索引中记录：

- 文件用途
- 维护范围
- 最近更新时间
- 下次任务应该先读哪个文件

## 15. 最小可用版本

如果不想维护复杂结构，至少保留以下五块：

```markdown
# AI 记忆

## 目标

## 规则

## 决策

## 风险

## 当前状态
```

## 16. 后续执行约定

后续当用户说“记一下”“形成记忆”“写入记忆宫殿”“按记忆宫殿更新”时，Agent 应执行：

1. 判断信息是否有长期价值。
2. 选择合适的记忆文件。
3. 按九区结构分类。
4. 使用记忆卡片格式写入。
5. 如有冲突，保留旧记录并标记替代关系。
6. 在出口区域写一条本次更新摘要。

当用户说“读取记忆”“按记忆做”“继续上次”时，Agent 应执行：

1. 先读相关记忆文件。
2. 说明读取到的关键约束。
3. 再继续任务。
4. 如果发现记忆过期，提醒并建议更新。

## 17. 新项目接入流程

当一个项目第一次启用 AI 记忆宫殿时，按以下步骤初始化。

### 17.0 十分钟初始化向导

如果时间有限，按以下最短路径初始化：

1. 复制 `ai-memory-template.md` 为目标项目的记忆文件。
2. 填写文件头部的项目类型、主要产物、关键风险、验证方式和接续难点。
3. 在“入口”写一条项目目标记忆。
4. 在“门厅”写一条全局规则或用户偏好。
5. 在“地图”写核心目录、入口文件、测试命令和关键文档。
6. 在“工作台”写当前状态、阻塞点和下一步。
7. 在“出口”写下次启动提示。
8. 在总索引登记项目记忆文件路径。

最小验收标准：

- 后续 Agent 能在 3 分钟内知道项目目标、关键路径、当前状态和下一步。
- 每条已写记忆都有状态、区域、标签、触发场景、来源和更新时间。
- 如果有不确定信息，必须标为 `draft` 或 `#需复查`。

### 17.1 建立项目画像

先判断项目属于什么类型，再决定记忆重点。

项目画像至少包含：

- 项目类型：数据/前端/后端/全栈/移动端/文档/研究/运维/混合
- 主要产物：代码、SQL、接口、页面、报告、模型、配置、流程或知识库
- 关键风险：数据正确性、用户体验、接口兼容、性能、安全、交付一致性或知识过期
- 验证方式：测试命令、截图检查、数据抽样、人工验收、CI、对账或复盘
- 接续难点：上下文复杂、依赖多、状态易过期、路径分散、规则不稳定或多人并行

### 17.2 自动生成项目画像

当用户没有直接提供项目画像时，Agent 可以从本地上下文生成草案。

优先读取：

- 目录结构：顶层目录、`src`、`app`、`docs`、`sql`、`scripts`、`tests`、`config`、`deploy`
- 项目元数据：`package.json`、`pom.xml`、`build.gradle`、`pyproject.toml`、`requirements.txt`、`go.mod`
- 文档入口：`README.md`、`docs/`、`doc/`、`*.md`
- 验证入口：测试目录、CI 配置、脚本命令、Makefile
- 数据入口：SQL 文件、迁移文件、数据字典、表结构文档

推断规则：

- 出现大量 SQL、数据同步脚本、表结构或调度脚本：倾向 `数据`
- 出现页面、组件、路由、样式、构建配置：倾向 `前端` 或 `移动端`
- 出现控制器、服务、数据库迁移、接口文档：倾向 `后端`
- 前端和后端结构同时明显存在：倾向 `全栈`
- 主要是 Markdown、报告、方案和研究材料：倾向 `文档` 或 `研究`
- 主要是部署、同步、监控、初始化脚本：倾向 `运维`

置信度规则：

- `high`：至少两个独立证据指向同一项目类型
- `medium`：一个强证据或多个弱证据
- `low`：只有目录名、用户上下文或 Agent 推测

写入规则：

- 自动推断的画像必须标明 `画像来源` 和 `画像置信度`
- `low` 置信度画像不得作为长期规则，只能设为 `draft`
- 如果画像会影响任务范围、技术选型或验证方式，应请求用户确认

### 17.3 自动画像命令清单

Agent 生成项目画像时，优先使用只读命令，避免改动项目。

推荐命令：

```bash
pwd
find . -maxdepth 2 -type d | sort | head -120
rg --files -g 'README*' -g '*.md' -g 'package.json' -g 'pom.xml' -g 'build.gradle' -g 'pyproject.toml' -g 'requirements.txt' -g 'go.mod' -g 'Makefile' -g '.github/**' | head -120
rg --files -g '*.sql' -g '*.py' -g '*.js' -g '*.ts' -g '*.tsx' -g '*.java' -g '*.go' -g '*.yaml' -g '*.yml' | head -160
rg -n "test|build|start|dev|deploy|migrate|sync|schedule|cron|api|route|controller|service|table|schema" . -g '!*node_modules*' -g '!*.lock' | head -120
```

读取顺序：

1. 先看目录，判断项目边界和候选子项目。
2. 再看元数据文件，判断技术栈和命令。
3. 再看 README 或 docs，提取项目目标和使用方式。
4. 再看文件类型分布，判断主要产物。
5. 最后用关键词搜索验证推断。

写入时必须区分：

- `观察事实`：命令真实看到的目录、文件和配置
- `推断判断`：Agent 对项目类型、风险和验证方式的判断
- `待确认问题`：证据不足但会影响后续行动的问题

### 17.4 项目登记自动更新

新项目创建或发现项目记忆文件后，应更新总索引的“项目记忆登记”。

登记字段建议包含：

```markdown
### 项目名称

- 状态：active/draft/archived
- 项目类型：
- 画像置信度：high/medium/low
- 记忆文件：
- 项目路径：
- 当前阶段：
- 最近更新时间：
- 下次先读：
- 未决问题：
```

自动更新规则：

- 新增项目记忆文件后，必须登记。
- 项目路径、记忆文件、当前阶段变化后，应更新登记。
- 画像置信度为 `low` 时，登记状态使用 `draft`。
- 项目归档后，登记状态改为 `archived`，不删除历史登记。
- 如果多个项目共享一个记忆文件，应在登记中写明范围，避免误读。

### 17.5 探测结果到记忆卡片的映射

自动画像完成后，Agent 应把观察结果转换为少量高价值记忆卡片。

映射规则：

| 探测结果 | 写入区域 | 记忆重点 |
| --- | --- | --- |
| README、需求文档、项目名 | 入口 | 项目目标、维护对象、成功标准 |
| 用户偏好、协作规范、代码规范 | 门厅 | 稳定规则、禁止事项、输出偏好 |
| 目录结构、入口文件、配置文件 | 地图 | 核心目录、入口代码、关键文档、配置位置 |
| 表结构、领域术语、指标口径 | 展厅 | 核心概念、实体关系、业务口径 |
| package scripts、Makefile、部署脚本、数据脚本 | 工坊 | 可重复执行的命令和流程 |
| 架构说明、选型文档、历史方案 | 档案室 | 决策、替代方案、原因和影响 |
| TODO、FIXME、异常日志、失败记录、危险命令 | 警示墙 | 风险触发条件、影响、规避和验证 |
| 当前分支、未完成任务、用户最新要求 | 工作台 | 当前状态、阻塞点、下一步 |
| 最近总结、交接说明、未决问题 | 出口 | 下次启动提示、待确认问题 |

首批卡片生成顺序：

1. 入口：项目目标或画像草案。
2. 地图：核心目录、入口文件和验证命令。
3. 工作台：当前状态和下一步。
4. 警示墙：探测到的高风险或待确认问题。
5. 出口：下次启动提示。

控制规则：

- 首次自动生成不要超过 8 条卡片。
- 只把明确观察到的内容写为 `active`。
- 推断内容写为 `draft`，并加 `#需复查`。
- 不把文件列表机械复制进记忆，只记录后续会影响行动的路径。
- 发现多个子项目时，先写总览地图，再决定是否拆分记忆文件。

### 17.6 初始化问答

当目录证据不足，或画像置信度为 `low` 时，Agent 应优先问少量关键问题。

推荐问题：

1. 这个项目的主要产物是什么？
2. 当前最重要的成功标准是什么？
3. 哪些规则、数据口径或用户偏好不能改？
4. 后续接手时最容易踩坑的地方是什么？
5. 这次任务完成后应该用什么方式验收？

问答写入规则：

- 用户明确回答的内容优先级高于 Agent 推断
- 未回答的问题记录到“出口”或“工作台”
- 如果用户回答改变了自动画像，应把旧画像标为 `superseded`

### 17.7 选择记忆粒度

按项目复杂度选择一种形态：

- 单文件：适合小项目、临时任务、个人脚本，使用 `doc/ai-memory.md`
- 多文件：适合长期项目、多人协作、产品研发，使用 `memory/` 目录
- 总控文件加领域文件：适合多项目组合，使用一个总索引指向各项目记忆

### 17.8 建立入口记忆

至少写入三条记忆：

- 项目或主题目标
- 当前阶段和成功标准
- 用户或团队的稳定偏好

### 17.9 建立地图记忆

首次接入必须记录：

- 核心目录
- 主要文档
- 入口代码
- 测试或验证命令
- 数据或配置位置

### 17.10 建立工作台记忆

当前任务必须写清：

- 已知背景
- 已完成事项
- 正在处理事项
- 阻塞点
- 下一步

### 17.11 建立出口记忆

每次较大更新后必须留下下次启动提示，让后续 Agent 不需要重新猜上下文。

## 18. 项目类型适配

不同项目的记忆重点不同。Agent 接入项目后，应先按项目画像选择优先区域。

### 18.1 数据项目

优先记忆：

- 数据来源、表结构、字段口径和分区规则
- SQL、脚本、调度、依赖链路和执行顺序
- 对账方式、抽样规则、异常数据和历史修复
- 性能风险、幂等风险、重复写入风险

优先读取区域：地图、展厅、工坊、警示墙、档案室。

### 18.2 前端项目

优先记忆：

- 页面结构、路由、组件边界和状态流
- 设计规范、交互约束、响应式规则和视觉验收方式
- API 契约、Mock 数据和用户关键路径
- 浏览器验证结果、截图问题和移动端风险

优先读取区域：门厅、地图、展厅、工坊、警示墙。

### 18.3 后端项目

优先记忆：

- 服务边界、接口契约、数据库模型和鉴权规则
- 迁移脚本、配置项、任务队列和外部依赖
- 测试命令、兼容性要求、错误码和回滚方案
- 性能、安全、并发和事务风险

优先读取区域：地图、档案室、工坊、警示墙、工作台。

### 18.4 全栈或移动端项目

优先记忆：

- 前后端契约、端侧限制、版本兼容和发布流程
- 关键用户流程、状态同步、离线或弱网行为
- 多端差异、构建命令和验收环境

优先读取区域：入口、门厅、地图、工坊、警示墙、工作台。

### 18.5 文档、研究或方案项目

优先记忆：

- 读者对象、叙事目标、术语口径和引用来源
- 章节结构、论证链路、待验证事实和版本变化
- 输出格式、审阅意见和未决问题

优先读取区域：入口、展厅、档案室、工作台、出口。

### 18.6 运维或自动化项目

优先记忆：

- 环境、权限、命令、配置和依赖版本
- 执行前检查、失败恢复、日志位置和告警规则
- 不可逆操作、外部影响和人工确认点

优先读取区域：门厅、地图、工坊、警示墙、出口。

## 19. 成熟度标准

一个记忆宫殿是否成熟，用以下标准判断。

### 19.1 L0：无记忆

没有结构化记忆，只依赖当前对话。

### 19.2 L1：有记录

有 Markdown 记录，但内容偏流水账，缺少分类、状态和触发场景。

### 19.3 L2：可检索

记忆按区域组织，包含标题、标签、状态、触发场景和更新时间。

### 19.4 L3：可接续

新 Agent 读取后能知道目标、规则、当前状态、关键文件和下一步。

### 19.5 L4：可治理

能处理冲突、过期、替代、归档和多 Agent 协作，记忆不会无限膨胀。

### 19.6 L5：可迁移

这套记忆能被复制到新项目，仍能快速初始化、演进和复盘。

## 20. 质量检查清单

每次更新记忆后，用以下清单检查。

### 20.1 内容质量

- 是否只记录未来会影响行动的信息
- 是否删除了无复用价值的过程描述
- 是否把长内容压缩成事实、判断、行动
- 是否能被没有参与当前对话的 Agent 理解

### 20.2 结构质量

- 是否放入了正确区域
- 是否有状态、标签、触发场景、来源和更新时间
- 是否和已有记忆重复
- 是否需要拆成多条记忆

### 20.3 检索质量

- 标题是否能被 `rg` 搜到
- 标签是否稳定
- 文件路径是否准确
- 下次任务开始时是否知道先读哪里

### 20.4 治理质量

- 过期内容是否标为 `deprecated` 或 `archived`
- 被替代内容是否标为 `superseded`
- 冲突是否记录了取舍原因
- 工作台是否只保留当前状态，不沉积长期历史

### 20.5 安全质量

- 是否避免写入密码、密钥、令牌和敏感个人信息
- 是否把推理结论标为 `draft`
- 是否区分用户明确要求和 Agent 推断

## 21. 冲突解决与确认机制

记忆冲突不可直接覆盖。Agent 必须先判断冲突等级，再决定自动处理、标记待确认，或请求用户确认。

### 21.1 来源优先级

当不同来源的信息冲突时，默认优先级如下：

1. 用户在当前对话中的明确要求
2. 用户历史明确偏好或规则
3. 项目内最新正式文档
4. 当前代码、配置、测试结果或数据观察
5. 已归档的旧决策
6. Agent 推理结论

如果更低优先级的信息看起来更可信，不能直接覆盖高优先级记忆，应标记为 `draft` 或 `#需复查`。

### 21.2 时间优先级

同一来源、同一主题冲突时，默认采用更新时间更新的记忆。

例外情况：

- 旧记忆是长期规则，新记忆只是临时状态
- 新记忆来源不明确
- 新记忆会改变用户偏好、技术方向、数据口径或外部可见行为

### 21.3 必须用户确认的变更

以下变更不能由 Agent 静默确定：

- 修改用户长期偏好
- 推翻项目目标或成功标准
- 改变数据口径、财务口径、权限规则或安全规则
- 改变技术选型、架构边界或发布策略
- 删除仍可能影响未来任务的决策记录
- 记录或处理敏感信息
- 执行不可逆或外部可见操作的规则变更

### 21.4 可以自动处理的变更

以下变更通常可以自动处理：

- 修正明显过期的工作台状态
- 标记已被新记录替代的旧交接
- 合并重复标签或重复表述
- 补充文件路径、更新时间或触发场景
- 将推理结论标为 `draft`

### 21.5 冲突记录格式

```markdown
### C-YYYYMMDD-序号：冲突标题

- 状态：draft/active/resolved
- 冲突主题：
- 旧记忆：
- 新信息：
- 来源对比：
- 默认判断：
- 需要用户确认：是/否
- 处理结果：
- 更新时间：YYYY-MM-DD
```

### 21.6 裁决后处理

确认新规则后：

- 新记忆设为 `active`
- 被替代记忆设为 `superseded`
- 保留替代关系和原因
- 在出口区域写入交接摘要

无法确认时：

- 新信息设为 `draft`
- 标签加入 `#需复查`
- 不改变原有 `active` 规则

## 22. 记忆压缩规则

当记忆文件变长时，按以下方式压缩。

### 22.1 合并重复

同类规则合并为一条原则，保留最新来源和更新时间。

### 22.2 归档历史

已完成的工作台历史移动到归档文件，当前文件只保留最近状态和可复用结论。

### 22.3 决策不压掉原因

决策记录可以压缩过程，但必须保留：

- 当时选择
- 替代方案
- 选择原因
- 后续影响

### 22.4 风险不压掉触发条件

风险记录必须保留“什么时候会发生”，否则后续 Agent 无法使用。

## 23. 记忆审计节奏

记忆宫殿需要定期审计，避免变成过期记录的堆积。

### 23.1 轻量审计

触发时机：

- 每次自动巡检
- 每次较大任务完成
- 每次用户要求“继续上次”或“按记忆做”

检查内容：

- 工作台是否仍是当前状态
- 出口交接是否指向下一步
- 是否有明显过期、重复或冲突记忆
- 是否需要更新成熟度

### 23.2 阶段审计

触发时机：

- 项目阶段切换
- 记忆卡片超过 30 条
- 多 Agent 并行开始或结束
- 重要架构、数据口径或产品方向变化

检查内容：

- 是否需要拆分文件
- 决策记录是否保留原因
- 风险记录是否仍有效
- 地图中的路径和命令是否还能使用
- 归档内容是否可以迁移或删除

### 23.3 迁移审计

触发时机：

- 复制到新项目
- 总结为通用模板
- 从单项目升级为多项目体系

检查内容：

- 是否删除了项目专属路径
- 是否保留了可复用流程
- 标签是否仍适用
- 成熟度是否需要重新评估

## 24. 自动巡检报告格式

自动巡检应使用固定格式，便于用户快速判断是否有价值。

### 24.1 有改动时

```markdown
本轮优化完成：

- 改动文件：
- 改动主题：
- 解决的问题：
- 当前成熟度：
- 下一轮观察点：
```

### 24.2 无改动时

```markdown
本轮巡检未发现必须修改的问题。

- 当前状态：
- 成熟度判断：
- 暂不改动原因：
- 下一轮观察点：
```

### 24.3 通知规则

应该通知用户：

- 修改了记忆宫殿规范、模板或索引
- 发现影响后续任务的冲突、过期或风险
- 成熟度发生变化
- 需要用户确认长期规则

可以静默：

- 只是确认文件仍然健康
- 没有发现新缺口
- 只做内部观察且不需要用户行动

## 25. 迁移规则

把记忆宫殿迁移到新项目时，不直接复制所有内容。

推荐流程：

1. 复制模板。
2. 复制通用规则。
3. 删除项目特有路径和状态。
4. 保留可迁移流程、质量标准和风险检查。
5. 重新建立入口、地图和工作台。
6. 在出口写明迁移来源和调整点。

## 26. 跨项目复用包

当需要把本体系带到新项目时，最小复用包包含：

- `ai-agent-memory-palace.md`：完整规范。
- `ai-memory-template.md`：新项目记忆模板。
- `ai-memory-index.md`：总索引和项目登记入口。
- `ai-memory-example-project.md`：填写示例。

复用顺序：

1. 复制模板到目标项目。
2. 用十分钟初始化向导填写头部画像和首批卡片。
3. 如果用户没有提供画像，使用自动画像命令清单生成草案。
4. 用探测结果映射规则生成入口、地图、工作台、警示墙和出口卡片。
5. 在总索引登记项目。
6. 用质量检查和确认检查做一次轻量审计。

复用边界：

- 复制结构，不复制旧项目事实。
- 复制通用规则，不复制项目专属路径。
- 复制流程，不复制用户未确认偏好。
- 复制示例时必须标明来源，避免误当真实状态。

## 27. 停止自动优化条件

当记忆宫殿达到以下条件时，应停止周期性自动扩展，避免规范膨胀：

- 已覆盖九区结构、写入、检索、生命周期、冲突、审计、迁移和初始化。
- 已有模板、索引和示例文件。
- 新项目能在 10 分钟内初始化到 L3。
- 复杂项目能通过标签、拆分、审计和冲突机制推进到 L4。
- 体系能复制到新项目并重新建立画像、地图和工作台，即达到 L5。
- 连续两轮优化只是在补细枝末节，而不是解决真实缺口。

停止后保留的维护方式：

- 用户明确要求时再继续优化。
- 真实项目接入暴露缺口时再更新规范。
- 审计发现过期、冲突或安全问题时再更新。
- 不因“还能想到新规则”而继续扩展。

停止汇报格式：

```markdown
自动优化已收束：

- 当前成熟度：
- 已完成能力：
- 停止原因：
- 后续触发条件：
```

## 28. 自动优化协议

当用户要求“自动优化到成熟”时，Agent 每轮按以下顺序执行：

1. 读取当前记忆宫殿规范和模板。
2. 用成熟度标准判断当前等级。
3. 找出最影响跨项目复用的缺口。
4. 只做一组高价值改动，避免一次性膨胀。
5. 更新文档或模板。
6. 在汇报中说明本轮改进、当前成熟度和下一轮观察点。
7. 如果没有必要改动，按自动巡检报告格式说明静默原因或当前状态。
8. 如果满足停止自动优化条件，应关闭周期性自动化，并说明后续触发条件。

默认优化目标：

- 让新项目接入更快
- 让后续 Agent 更容易检索
- 让记忆更少过期和冲突
- 让规则能迁移到不同技术栈、不同领域、不同任务类型

## 29. 示例

```markdown
### M-20260511-001：后续记忆采用 AI 记忆宫殿结构

- 状态：active
- 区域：门厅
- 标签：#记忆宫殿 #Agent协作 #Markdown

- 记忆内容：后续记忆不使用普通流水账，而使用“入口、门厅、地图、展厅、工坊、档案室、警示墙、工作台、出口”的九区结构，并以记忆卡片形式写入 Markdown。
- 使用方式：先判断记忆价值，再分类写入；每条记忆必须包含状态、区域、标签、触发场景、内容、使用方式、来源和更新时间。
- 来源：用户明确要求
- 更新时间：2026-05-11
```
