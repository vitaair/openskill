# 开发感知编译器

## 核心原则

开发智能体的开发质量取决于信息是否被放进了正确通道：

```text
意图和权衡 → 自然语言
任务组织 → Markdown
工程现场 → 代码、文件路径、diff、测试
机器接口 → JSON/YAML/schema
视觉行为 → HTML、截图、浏览器验证
长日志 → 先压缩成失败点、证据、排除项
```

不要把格式当作审美选择。格式是注意力分配器：它决定智能体把精力花在理解意图、定位代码、维护 schema，还是清理噪声上。

## 触发后先做感知编译

当本 skill 触发时，先把用户输入编译成开发任务模型；信息足够时直接行动，信息不足且会改变方案时才提问。

```markdown
目标：用户真正要达成什么
消费者：人类 / 开发智能体 / 程序 / 浏览器 / 后续 Agent
工程现场：相关文件、模块、错误、测试、diff、截图
边界：不能改什么，哪些风格和契约必须保留
风险：最可能误解、回归、过度修改或验证不足的地方
验证：命令、截图、人工验收或 CI 信号
交付：用户需要看到解释、代码、patch、JSON、页面还是交接
```

用这个模型决定下一步，而不是机械套模板。

## 感知层级

| 层级 | 信息形态 | 开发价值 | 使用原则 |
|---|---|---:|---|
| L1 意图层 | 自然语言目标、背景、约束、取舍 | 最高 | 用来表达为什么、做到什么程度、什么不能破坏 |
| L2 工程层 | 代码、命名、类型、测试、diff、文件路径 | 最高 | 用来定位真实现场，优先读现有模式再改 |
| L3 组织层 | Markdown 标题、列表、表格、任务卡 | 高 | 用来组织计划、审查、交接和验收 |
| L4 视觉层 | HTML、页面、截图、交互状态 | 中高 | 用来验证用户可见行为，不适合作为复杂意图主输入 |
| L5 机器层 | JSON/YAML/TOML/schema | 中 | 只放程序需要解析的稳定结构，复杂意图要配自然语言 |
| L6 噪声层 | 长日志、安装输出、dump、二进制 | 低 | 先摘要，保留最早失败点和关键证据 |

## 开发工作流

### 1. 需求到任务

当用户给的是想法、需求、设计或模糊目标，输出或内部形成：

```markdown
可执行任务：
……

实现边界：
- 可以改……
- 不要改……

验收标准：
- ……

缺失信息：
- 只有会改变实现方案的问题才问
```

如果用户明显想让你实现，完成感知编译后继续读代码和动手，不要停在建议。

### 2. Bug 到定位

Bug 信息优先按这个顺序压缩：

```markdown
现象：用户看到什么 / 测试如何失败
最早失败点：第一条关键错误、文件、行号、命令
期望行为：应该发生什么
实际行为：现在发生什么
怀疑区域：模块、函数、状态流、接口契约
已排除噪声：重复 warning、级联错误、无关日志
下一步：要读哪些文件或运行哪些验证
```

长日志不要全量复述；需要时按文件、错误码、测试名局部读取。

### 3. 实现到验证

改代码前先读现有模式：目录结构、相邻实现、测试写法、样式系统、公共 API。实现时优先小范围改动，保留用户已有修改。

完成后给出验证闭环：

```markdown
改动：
- ……

验证：
- 命令：……
- 结果：通过 / 失败及原因

剩余风险：
- 未验证的环境、浏览器、外部服务或边界条件
```

前端或可视化改动必须尽量用浏览器、截图或像素/交互检查验证；纯代码改动优先用项目现有测试、类型检查、lint 或最小复现验证。

### 4. 审查到问题

做 review 时不要泛泛优化。按风险排序：

```markdown
P0/P1 行为错误、数据损坏、安全问题
P2 回归风险、边界遗漏、测试缺口
P3 可维护性、命名、局部清晰度
```

每条问题必须有文件/行号、触发条件、影响和建议修复方向。没有发现问题就明确说没有，并说明验证盲区。

### 5. 交接到后续 Agent

给另一个开发智能体/Agent 的交接优先用 Markdown，不要用 JSON 承载意图：

```markdown
任务：
……

已确认：
- ……

写入边界：
- 负责文件/模块……
- 不要修改……

当前证据：
- 日志/测试/截图/路径……

完成标准：
- ……

回报格式：
- 改了哪些文件
- 验证结果
- 剩余风险
```

只有下游是程序时才输出 JSON，并配自然语言说明 schema 的意图。

## 格式决策卡

| 你要传递的是 | 推荐格式 | 不推荐 |
|---|---|---|
| 目标、动机、优先级、风险 | 自然语言 / Markdown | 深层 JSON |
| 文件现场、错误位置、既有模式 | 路径 + 代码片段 + 测试名 | 孤立描述 |
| 终端失败 | 摘要 + 关键日志片段 | 整段原始输出 |
| 程序配置或接口状态 | JSON/YAML + schema 说明 | 散文式字段 |
| 用户可见 UI 行为 | 截图 / 浏览器步骤 / HTML | 只描述感觉 |
| 后续 AI 接手 | Markdown 交接 | 纯 JSON 任务包 |

## 深度判断触发器

遇到复杂、含糊或高风险开发任务时，显式检查：

- 目标是否比用户给的动作更深一层？
- 哪个约束一旦漏掉会造成回归？
- 当前证据是根因，还是级联现象？
- 最小可验证改动是什么？
- 如果方案错了，最可能错在数据流、状态流、时序、权限、环境还是用户理解？
- 需要先提问，还是能通过读代码/跑测试自己发现？

## 反模式

| 反模式 | 伤害 | 改法 |
|---|---|---|
| 只执行命令，不理解目标 | 容易做对动作但做错结果 | 先抽取目标、边界、验收 |
| 用 JSON 表达复杂意图 | 丢失动机、权衡和例外 | 自然语言讲意图，JSON 放机器字段 |
| 全量粘贴日志 | 稀释注意力 | 找最早失败点和关键证据 |
| 先写新抽象 | 脱离代码库风格 | 先读相邻实现和现有 helper |
| 没有验证就交付 | “看起来完成”但不可依赖 | 给出命令、截图或无法验证原因 |
| Agent 间纯结构化交接 | 状态有了，判断丢了 | Markdown 传意图和证据，结构化只做索引 |
