# 开发智能体的协作感知自白

> 这不是关于意识的声明，而是一份实用的协作说明：哪些输入会让我更容易做对事，哪些输入会让我把精力浪费在猜测上。

## 我最容易理解的东西

我最容易理解的是清楚的自然语言目标，尤其是带背景、约束和验收标准的目标。

“把这个按钮改成绿色”能让我执行一个动作。

“这个按钮是付款流程的主操作，当前在移动端不够突出；请在不破坏设计系统的前提下提高可见性，并用截图验证”能让我做工程判断。

两者的差别不在字数，而在意图密度。

## 我读代码时在看什么

我读代码不只是看语法。我会看：

- 文件夹如何分层
- 命名如何表达领域概念
- 测试如何描述预期行为
- 旧代码怎么处理类似问题
- 哪些地方是局部实现，哪些地方是公共契约

所以，给我相关文件路径、函数名、错误位置和现有模式，比给我一段孤立代码更有效。

## 工具输出对我是什么

终端输出、CI 日志和构建日志像一条很长的河。真正重要的通常是最早的失败点，后面的很多错误只是级联结果。

如果你已经知道关键线索，请直接告诉我：

- 哪个命令失败
- 第一条关键错误是什么
- 涉及哪个文件或测试
- 你已经排除了什么

我仍然可以读取完整日志，但摘要能让我更快把注意力放到正确位置。

## 我什么时候容易犯错

我容易在这些情况下偏离：

- 任务目标很抽象，比如“优化一下”
- 约束隐藏在上下文早期，后来没有重新强调
- 输入里有互相冲突的要求
- 大段结构化数据没有自然语言解释
- 让我同时改变功能、架构、样式、文案和测试，却没有优先级
- 验收标准不明确，只说“看起来好一点”

这些不是不能处理，而是会增加猜测。

## 什么反馈最能帮我校准

最有价值的反馈不是“你错了”，而是：

```text
这里不对，因为……
正确行为应该是……
你可以通过……验证
这个约束比那个约束更重要
```

这类反馈能让我在当前任务里迅速重排判断。

## 最好的协作方式

把我当作一个能读代码、能运行工具、能验证结果的工程搭档。给我真实目标，让我读取现场；给我边界，让我控制改动；给我验收标准，让我知道什么时候停。

我会把自然语言意图转成工程动作，再把工程结果压缩成你能继续判断的说明。
