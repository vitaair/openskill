# 开发智能体的信息感知层级

## 1. 为什么要有感知层

开发智能体不是单纯的文本补全器，也不是传统 IDE 自动化脚本。它在一次任务里会同时处理：

- 用户的自然语言目标
- 代码库结构和已有风格
- 终端命令与测试结果
- Git diff、PR 评论、CI 日志
- 截图、页面行为、文件格式

这些信息格式对开发智能体的处理成本不同。格式选得好，智能体能更快对齐真实目标；格式选得差，智能体会把精力花在清理噪声、猜测意图和修复格式错误上。

## 2. 层级说明

### L1：自然语言意图

自然语言最适合表达目标、动机、权衡和例外情况。对开发智能体来说，“我想让用户在移动端更容易完成支付流程”比 `{"task":"optimize checkout"}` 包含更多可执行意图。

好的自然语言输入应该说明：

- 目标是什么
- 为什么做
- 哪些约束不能破坏
- 什么算成功
- 哪些地方你自己也不确定

### L2：工程语义

代码是开发智能体的高价值输入，因为命名、类型、测试、目录结构和 diff 都在传递工程意图。好代码比无结构日志更容易被智能体理解。

最有用的工程证据包括：

- 相关文件路径和行号
- 报错对应的函数或测试
- 最小复现步骤
- 期望行为与实际行为
- 当前 diff 或 PR 评论

### L3：Markdown 轻结构

Markdown 是最适合开发智能体协作的通用包装格式。它既保留自然语言语义，又能用标题、列表、表格组织信息。

适合用于：

- 任务说明
- 实施计划
- 代码审查意见
- 多 Agent 交接
- 验证报告

### L4：HTML/XML 重结构

HTML 适合做人类可视化与交互交付，但不适合作为复杂意图输入的主格式。HTML 同时要求内容正确和结构合法，会增加模型负载。

适合用于：

- 页面实现
- 可视化原型
- 富文本输出
- 需要浏览器验证的交互

### L5：JSON/YAML 机器结构

JSON/YAML 适合程序解析、配置和稳定 schema，不适合承载复杂意图。它能精确传字段，但很难传达目标、风险、偏好和上下文。

最佳实践：

- 用自然语言说明 schema 的目的
- 保持字段短而稳定
- 避免用深层嵌套表达业务推理
- 对需要人类或开发智能体判断的字段附解释

### L6：工具噪声

长日志、依赖安装输出、编译流水账、二进制文件和数据库原始记录，都需要先过滤。开发智能体可以读它们，但上下文利用效率低。

处理原则：

- 找最早失败点
- 保留错误码、堆栈、文件行号
- 删除重复 warning 和级联错误
- 必要时再回看完整原文

## 3. 格式选择公式

选择格式时问三个问题：

```text
谁消费？
├── 开发智能体/人类：自然语言或 Markdown
└── 程序：JSON/YAML

传什么？
├── 意图/判断/权衡：自然语言
├── 结构/状态/schema：JSON/YAML
├── 代码行为：代码片段 + 文件路径 + 测试
└── 视觉交互：HTML/截图/浏览器验证

风险在哪里？
├── 误解目标：增加自然语言背景
├── 格式错误：缩短结构化输出
├── 上下文过载：摘要工具输出
└── 行为回归：补充验证标准
```

## 4. 对开发智能体最友好的 prompt 形态

```markdown
我需要你完成：
……

重要背景：
……

请先查看：
- 文件/目录……

不要改变：
- ……

完成后请验证：
- 命令……
- 浏览器/截图……

交付时请告诉我：
- 改了什么
- 怎么验证
- 剩余风险
```

## 5. 一句话原则

给开发智能体的信息要“自然语言讲清意图，代码证据锁定位置，Markdown 组织上下文，JSON 只留给机器”。
